# Preview Runtime Stability Design

## Goal

Improve Vibe Coding Studio app preview reliability for agent-generated full-stack apps without relying on tighter agent prompts.

The platform should:

1. Accept reasonable path and filename drift in generated frontend/backend projects.
2. Resolve runnable frontend/backend services from workspace contents instead of assuming fixed filenames.
3. Bootstrap runtime dependencies predictably inside the sandbox.
4. Publish preview readiness only when the relevant services are actually healthy.
5. When startup fails, return a structured diagnostic report to the engineer agent for one or more repair rounds instead of failing immediately.

This design focuses on platform/runtime behavior. It does not require changing the SWE base system prompt.

## Non-Goals

- Solving arbitrary frontend/backend API contract mismatches generated by the model.
- Replacing agent planning or todo orchestration with a full task graph engine in this iteration.
- Supporting every possible framework. This design targets the common Vibe Coding Studio path first: React/Vite frontend and Python backend, with extensible detection.

## Current Failure Pattern

The current preview pipeline has two distinct failure classes:

1. Platform/runtime failures
   - Backend command depends on tools not present in the sandbox, such as `uv`.
   - Backend dependencies are not bootstrapped automatically.
   - Preview can be marked ready even when backend healthcheck failed.
   - Runtime startup assumes more fixed structure than agent outputs guarantee.

2. Agent generation failures
   - Frontend and backend routes or payload shapes do not align.
   - Entry points are renamed without corresponding import/runtime fixes.

This design addresses the first class directly and improves recovery for the second by producing actionable diagnostics.

## Design Principles

### Resolve, then verify, then publish

The platform must not trust generated commands or filenames blindly. It should:

1. Resolve likely runtime model from workspace contents.
2. Attempt bootstrap and startup.
3. Verify healthchecks.
4. Publish preview only after verification.

### Accept naming drift, not execution ambiguity

The platform should tolerate:

- `main.py` vs `server.py` vs `app.py`
- `app/frontend` vs `frontend`
- `app/backend` vs `backend`

The platform should not accept:

- no discoverable backend entrypoint
- startup commands that cannot run in the sandbox and cannot be normalized
- missing healthcheck evidence for services presented as ready

### Diagnostics must be structured

Failure output should be machine-usable by the engineer agent, not just human-readable logs.

## End-to-End Target Flow

```text
user request
  -> engineer runtime
  -> agent writes workspace files + preview manifest
  -> preview resolver builds normalized runtime model
  -> sandbox bootstrap installs deps and starts services
  -> health verification runs
  -> if success: preview_ready
  -> if failure: structured diagnostics -> repair continuation round
  -> retry bootstrap once or twice
  -> final success or final failure with diagnostics
```

## Runtime Model

Add an internal normalized runtime model for preview startup.

```text
PreviewRuntimeModel
  frontend:
    root
    kind
    install_strategy
    start_strategy
    healthcheck_path
    base_path_support
  backend:
    root
    kind
    install_strategy
    start_strategy
    healthcheck_path
  source:
    manifest_confidence
    fallback_confidence
  diagnostics:
    warnings
```

### Purpose

This model decouples platform startup behavior from raw workspace filenames. The rest of the preview pipeline should operate on the normalized model, not directly on `.atoms/preview.json` or guessed filenames.

## Resolution Strategy

Resolution should happen in priority order.

### 1. Manifest-first resolution

Read `.atoms/preview.json` if present.

Use it as the highest-priority source for:

- explicit frontend/backend command
- explicit healthcheck path
- optional service presence

But the manifest should still be validated and normalized. A manifest command is input, not proof.

### 2. Convention fallback resolution

If the manifest is missing, incomplete, or suspicious, use convention-based discovery.

#### Frontend discovery

Candidate roots, in order:

1. `/workspace/app/frontend`
2. `/workspace/frontend`
3. `/workspace`

Confidence signals:

- `package.json` exists
- `vite.config.*` exists
- `src/main.*` exists
- build/dev script exists

#### Backend discovery

Candidate roots, in order:

1. `/workspace/app/backend`
2. `/workspace/backend`
3. `/workspace`

Confidence signals:

- `requirements.txt` or `pyproject.toml` exists
- Python files contain `FastAPI(` or `app = FastAPI(`
- common entry candidates exist: `main.py`, `server.py`, `app.py`

### 3. Entrypoint resolution

For Python backend candidates, resolve a preferred import target in this order:

1. explicit manifest command, if valid after normalization
2. `main:app`
3. `server:app`
4. `app:app`
5. first file containing a clear FastAPI app symbol

If only a file path is discovered, the platform should derive an import target relative to the backend root.

## Bootstrap Strategy

### Frontend bootstrap

Current behavior is mostly correct: detect frontend root, install Node dependencies if needed, and run the standard dev server through `start-dev`.

Keep this path, but drive it from the normalized runtime model instead of direct string matching only.

### Backend bootstrap

This is the main missing capability.

For Python backends:

1. Detect backend root.
2. Create or reuse a local virtual environment under the backend root, for example:
   - `/workspace/app/backend/.venv`
3. Install dependencies from:
   - `requirements.txt`, or
   - future `pyproject.toml` support
4. Start backend using the venv interpreter.

Preferred startup form:

```text
<backend_root>/.venv/bin/python -m uvicorn <import_target> --host 0.0.0.0 --port 8000
```

The platform should not require `uv` to exist in the sandbox for backend preview startup.

### Command normalization

Manifest commands should be normalized when possible.

Examples:

- `uv run uvicorn server:app ...` -> normalized to venv-based `python -m uvicorn server:app ...`
- `python3 -m uvicorn server:app ...` -> allowed if import target resolves

If normalization is impossible, emit a structured diagnostic rather than attempting arbitrary shell behavior repeatedly.

## Preview Readiness Semantics

Preview readiness should become service-aware.

### Frontend-only apps

It is valid to publish preview when:

- frontend is healthy
- no backend is declared or discovered

### Full-stack apps

It is valid to publish full preview readiness only when:

- frontend is healthy
- backend is healthy

If frontend is healthy but backend is not:

- do not emit a misleading all-green ready state
- emit a degraded preview state with explicit backend failure diagnostics

The UI may still show the frontend iframe, but it must show backend startup failure as a first-class runtime issue.

## Repair Loop

When bootstrap or verification fails, the platform should produce a structured failure report and feed it back into the engineer runtime as a repair continuation.

### Failure report shape

```json
{
  "service": "backend",
  "phase": "bootstrap",
  "reason_code": "missing_runtime_uv",
  "detected_root": "/workspace/app/backend",
  "detected_entrypoint": "server:app",
  "attempted_command": "uv run uvicorn server:app --host 0.0.0.0 --port 8000",
  "stdout_tail": "",
  "stderr_tail": "bash: uv: command not found",
  "suggested_fix": "Use a Python backend startup that works without uv, or ensure the backend imports resolve with the detected entrypoint."
}
```

### Repair policy

Recommended limits:

- one initial startup attempt
- up to two repair rounds
- final failure after the retry budget is exhausted

The repair prompt should be narrow and operational:

- what service failed
- what root and entrypoint were detected
- what exact command failed
- what stderr/stdout tail was observed
- what concrete repair is requested

This keeps the continuation focused on runtime repair instead of reopening the whole implementation task.

## Responsibilities by Layer

### `engineer_runtime`

Responsible for:

- deciding whether to request a repair continuation round
- enforcing retry budget
- surfacing structured diagnostics to the agent and UI

Not responsible for:

- low-level framework detection
- shell-level package bootstrap logic

### preview resolver service

New or expanded service responsible for:

- manifest parsing
- candidate root detection
- entrypoint resolution
- runtime model creation

### sandbox startup scripts

Responsible for:

- deterministic bootstrap from normalized runtime model
- frontend dependency install
- backend venv creation and dependency install
- launching frontend/backend processes

### preview gateway / runtime router

Responsible for:

- correct routing and readiness semantics
- not overstating readiness when only frontend succeeded

## Error Taxonomy

Introduce stable reason codes for repair and analytics.

Initial set:

- `manifest_invalid`
- `frontend_root_not_found`
- `backend_root_not_found`
- `backend_entrypoint_not_found`
- `missing_runtime_uv`
- `python_dependency_install_failed`
- `python_dependency_not_installed`
- `entrypoint_import_failed`
- `frontend_healthcheck_timeout`
- `backend_healthcheck_timeout`
- `service_start_command_failed`

These should be used in both runtime logs and structured repair payloads.

## Testing Strategy

### Unit tests

- manifest parsing and normalization
- frontend root fallback resolution
- backend root fallback resolution
- Python entrypoint detection
- command normalization from `uv run uvicorn ...` to venv launch
- failure report generation

### Integration tests

- frontend-only workspace with path drift
- full-stack workspace with `server.py` instead of `main.py`
- backend requiring dependency install from `requirements.txt`
- backend healthcheck failure should not produce full ready
- failed startup should trigger repair continuation payload generation

### Regression tests

Add a fixture matching the recent feedback-board failure pattern:

- frontend under `app/frontend`
- backend under `app/backend`
- backend entry file named `server.py`
- manifest command uses `uv run`

Expected platform behavior:

- backend runtime is normalized and bootstrapped without requiring `uv`
- or a precise repair report is produced
- but not a false-success full preview state

## Rollout Plan

### Phase 1

- introduce runtime model and resolver
- add backend venv bootstrap in sandbox
- tighten preview readiness semantics

### Phase 2

- add structured failure report generation
- add repair continuation round in engineer runtime

### Phase 3

- add broader framework detectors and telemetry on reason codes

## Trade-offs

### Why not only tighten prompts?

Prompt tightening is cheap but fragile. Runtime stability should not depend on the model remembering exact backend startup conventions every time.

### Why not force one rigid template?

Rigid templates simplify the platform but reject many otherwise runnable generations. The better balance is constrained runtime resolution plus verification.

### Why not support arbitrary shell commands fully?

Unbounded shell flexibility makes startup unpredictable and diagnostics noisy. Normalized startup strategies are more stable, safer, and easier to repair.

## Recommendation

Implement a platform-first preview runtime with:

- manifest-first resolution
- convention fallback detection
- backend venv bootstrap
- strict service readiness semantics
- structured repair diagnostics

This gives the platform the best chance of running agent-generated apps even when filenames and directories drift, while still failing in a way the agent can repair when runtime resolution is genuinely impossible.
